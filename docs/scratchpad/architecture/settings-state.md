# Settings & State Plumbing

Subsystem map for Handy (evolving into Murmur). Covers the settings schema, persistence via
tauri-plugin-store, the Rust command layer that mutates settings, the React/zustand stores that
mirror them, and how changes propagate in both directions. Verified against source on 2026-07-04.

## Purpose

All user configuration lives in a single JSON blob (`AppSettings`) persisted by
`tauri-plugin-store` under one key (`"settings"`) in one file (`settings_store.json`). Rust is
the source of truth: the frontend never touches the store file directly; every read goes through
the `get_app_settings` command and every write goes through a per-setting Tauri command that does
a whole-struct read-modify-write. The frontend keeps an optimistic in-memory mirror in a zustand
store and rolls back on command failure.

## Persistence

- Plugin: `tauri-plugin-store = "2.4.1"` (registered in `src-tauri/src/lib.rs` line ~746 with
  `tauri_plugin_store::Builder::default().build()`).
- File: `settings.rs (SETTINGS_STORE_PATH)` = `"settings_store.json"`, resolved through
  `portable.rs (store_path)`:
  - Normal install: relative path -> plugin resolves to the app data dir. On macOS:
    `~/Library/Application Support/com.pais.handy/settings_store.json` (bundle id from
    `src-tauri/tauri.conf.json`).
  - Portable mode (marker file `portable` containing `"Handy Portable Mode"` next to the exe):
    `<exe_dir>/Data/settings_store.json`. Detected once at startup by `portable.rs (init)`,
    cached in a `OnceLock`.
- Layout: the store contains a single top-level key `"settings"` whose value is the entire
  serialized `AppSettings` struct. There are no per-setting store keys.
- Write behavior: `store.set(...)` triggers the plugin's auto-save with a **100 ms debounce**
  (plugin default; verified in tauri-plugin-store 2.4.2 source, `store.rs`). Auto-save errors are
  discarded inside the plugin (`let _ = self.trigger_auto_save()`), so a failed disk write is
  invisible to both Rust and the UI. The store is also flushed on graceful exit.
- The frontend has `@tauri-apps/plugin-store` in `package.json` but **never imports it**; all
  access is via commands.

## Rust API surface (settings.rs)

- `settings.rs (load_or_create_app_settings)` — startup path. Loads `"settings"`, parses to
  `AppSettings`, runs `apply_settings_migrations`, merges any missing default bindings, runs
  `ensure_post_process_defaults`, writes back if anything changed. On parse failure it logs
  `warn!` and **replaces the whole store with defaults** (user config, including API keys, is
  lost with no user-facing feedback).
- `settings.rs (get_settings)` — used by every command. Same load+migrate+defaults pipeline;
  reads can therefore write to disk (migrations are idempotent so this converges). Same
  reset-to-defaults behavior on parse failure.
- `settings.rs (write_settings)` — serializes the whole struct into `"settings"`.
- `settings.rs (get_bindings / get_stored_binding / get_history_limit / get_recording_retention_period)`
  — convenience readers. `get_stored_binding` does `bindings.get(id).unwrap()` and **panics** on
  an unknown id.
- All three store accessors call `.expect("Failed to initialize store")` — a store init failure
  is a process panic.

### Migrations (`settings.rs (apply_settings_migrations)`, schema version constant `CURRENT_SETTINGS_SCHEMA_VERSION = 1`)

Run on every load/read against the *raw* stored JSON:

1. Missing `onboarding_completed` -> set true iff `selected_model` non-empty (pre-existing users
   skip onboarding).
2. Missing `whats_new_last_seen_version` -> blank it (upgraders see current release notes; fresh
   installs stamp `CARGO_PKG_VERSION` via the serde default).
3. `settings_schema_version < 1` -> if `transcribe_gpu_device > 0`, reset it to `-1` and
   `transcribe_accelerator` to `Auto` (the value's meaning changed from UI ordinal to
   transcribe.cpp registry index), then stamp version 1.
4. Missing `overlay_style` -> derive from raw `overlay_position`: stored `"none"` -> `None`,
   anything else -> `Live`. (`OverlayPosition` no longer has `none`; a serde alias folds legacy
   `"none"` onto `Bottom` so old stores still deserialize.)

`settings.rs (ensure_post_process_defaults)` additionally back-fills newly added providers,
missing API-key/model map entries, and syncs `supports_structured_output` flags.

## bindings.ts generation (tauri-specta)

In `src-tauri/src/lib.rs (run)`:

- `Builder::<tauri::Wry>::new().commands(collect_commands![...]).events(collect_events![...])`
  registers ~110 commands and exactly 3 typed events (`managers::history::HistoryUpdatePayload`,
  `managers::transcription::StreamTextEvent`, `managers::transcription::StreamPhaseEvent`).
- Export is gated `#[cfg(debug_assertions)]`: bindings are (re)written to `../src/bindings.ts`
  **only when a debug build runs** (`Typescript::default().bigint(BigIntExportBehavior::Number)`,
  so Rust `u64`/`usize` become TS `number`). Release builds never regenerate the file; the
  committed file must be kept in sync by running the dev app.
- Commands returning `Result<T, String>` surface in TS as
  `Promise<Result<T, string>>` = `{ status: "ok"; data: T } | { status: "error"; error: string }`.
  Infallible commands surface as plain `Promise<T>`.
- The typed `events` export covers only the 3 specta events (TS names `historyUpdatePayload`,
  `streamPhaseEvent`, `streamTextEvent`; wire names `history-update-payload`,
  `stream-phase-event`, `stream-text-event`). Every other event in the app is an **untyped
  string event** via `app.emit(...)` / `listen(...)` — no compile-time payload checking.

## Runtime Flow

### Startup (Rust)

1. `portable.rs (init)` detects portable mode (marker file + magic string, with a legacy
   empty-marker migration path).
2. `lib.rs (run)` builds the specta builder, exports bindings (debug only), registers plugins
   (store, log, autostart, ...).
3. In setup, `settings.rs (load_or_create_app_settings)` loads/creates settings and applies
   migrations; managers (audio, transcription, model, history) are constructed against it.
4. `lib.rs` applies startup side effects: hides tray if `!show_tray_icon`, enables/disables
   autostart from `autostart_enabled`, listens (Rust-side) to `model-state-changed` to refresh
   the tray menu.
5. Shortcuts are NOT registered at startup on the normal path; the frontend calls
   `initialize_shortcuts` after onboarding/permission checks (`commands/mod.rs
   (initialize_shortcuts)` is idempotent via the `ShortcutsInitialized` marker state).
   `shortcut/mod.rs (init_shortcuts)` picks the backend from `keyboard_implementation`; if
   HandyKeys init fails it **persists a fallback to Tauri** into settings.

### Frontend init (React)

1. Any component calling `useSettings()` (`src/hooks/useSettings.ts`) triggers
   `settingsStore.initialize()` once (while `isLoading`).
2. `initialize` runs in parallel: `getDefaultSettings` (platform-correct defaults for reset
   buttons), `getAppSettings` (full settings; `refreshSettings` normalizes
   `selected_microphone`/`clamshell_microphone`/`selected_output_device` from `null` to
   `"Default"` for the UI), `checkCustomSounds`. It then subscribes to `model-state-changed`
   and refetches settings whenever it fires (backend is source of truth; e.g. model switch also
   flips `onboarding_completed`).
3. Audio device enumeration is deliberately NOT part of initialize: `src/App.tsx` calls
   `refreshAudioDevices()`/`refreshOutputDevices()` plus `initializeEnigo()` and
   `initializeShortcuts()` only after onboarding is done, to avoid triggering macOS
   microphone/accessibility permission dialogs early.
4. `src/App.tsx (checkOnboardingStatus)` reads `onboarding_completed` via `getAppSettings` and,
   on macOS, checks accessibility + microphone permissions
   (`tauri-plugin-macos-permissions-api`) to decide whether to show onboarding.

### Changing a setting (frontend -> Rust)

1. UI component calls `updateSetting(key, value)` on the store.
2. `settingsStore.ts (updateSetting)`: marks `isUpdating[key]`, applies the value optimistically
   to local state, then looks up `settingUpdaters[key]` — a map from settings key to the
   generated command (e.g. `push_to_talk` -> `commands.changePttSetting`). No entry and not
   `bindings`/`selected_model` -> `console.warn("No handler for setting: ...")` and the value is
   **only changed in memory** (silently diverges from disk).
3. The Rust command (mostly in `shortcut/mod.rs`) does `get_settings` -> mutate one field ->
   `write_settings`, plus any live side effect:
   - `change_overlay_position_setting` / `change_overlay_style_setting` -> reposition overlay
     window (`utils::update_overlay_position`, `shortcut/mod.rs:555, 581`); only
     `change_overlay_style_setting` also updates the overlay-enabled cache
     (`overlay.rs (update_overlay_enabled_cache)`, called at `shortcut/mod.rs:578`) —
     `change_overlay_position_setting` deliberately doesn't (per the comment at
     `shortcut/mod.rs:552-555`, enablement is owned by `overlay_style`; position only toggles
     Top/Bottom).
   - `change_debug_mode_setting` -> `WEBVIEW_LOG_STREAMING` atomic + emits `settings-changed`.
   - `change_autostart_setting` -> `app.autolaunch().enable()/disable()` (errors ignored).
   - `change_post_process_enabled_setting` -> registers/unregisters the
     `transcribe_with_post_process` shortcut (errors ignored).
   - `change_app_language_setting` -> `tray::update_tray_menu`.
   - `change_show_tray_icon_setting` -> `tray::set_tray_visibility`.
   - `change_transcribe_accelerator_setting` / `change_ort_accelerator_setting` /
     `change_transcribe_gpu_device` -> `save_accelerator_and_reload_next_use` marks the
     transcription engine for reload on next use.
   - `commands/mod.rs (set_log_level)` -> updates the `FILE_LOG_LEVEL` atomic then persists.
   - `commands/audio.rs (update_microphone_mode / set_selected_microphone /
     set_selected_output_device / set_clamshell_microphone)` -> persist then poke
     `AudioRecordingManager`.
4. `store.set("settings", ...)` schedules the debounced (100 ms) disk write.
5. On command error the store rolls the optimistic value back and logs to console (no toast).

### Shortcut bindings

- `settingsStore.ts (updateBinding)` -> `shortcut/mod.rs (change_binding)`: rejects empty
  bindings, unregisters old, validates for the active keyboard implementation, registers new,
  persists. Returns `BindingResponse { success, binding, error }`; the store checks both the
  outer `Result` and `success`, rolls back, and rethrows so `ShortcutInput` components can show
  the failure. `cancel` is special-cased (dynamically registered only while recording), so it is
  persisted without register/unregister.
- `suspend_binding` / `resume_binding` temporarily unregister/register while the user is
  recording a new hotkey in the UI.
- `change_keyboard_implementation_setting` (called directly from
  `src/components/settings/debug/KeyboardImplementationSelector.tsx`, bypassing
  `settingUpdaters`) unregisters everything, persists the new impl, re-validates every binding
  (resetting incompatible ones to defaults, persisted), and emits `settings-changed` with
  `reset_bindings`.

### Rust-initiated settings changes (Rust -> frontend)

There is **no generic typed settings event**. Two mechanisms exist:

1. `model-state-changed` (untyped): emitted by `commands/models.rs (switch_active_model)` and
   several places in `managers/transcription.rs` (load start/complete/failed, unload,
   selection_changed). `settingsStore` refetches all settings on it; `modelStore`,
   `ModelSelector`, and `App.tsx` also listen. Rust itself listens in `lib.rs` to refresh the
   tray. This is the de-facto "settings changed behind your back" signal.
2. `settings-changed` (untyped, payload `{ setting, value, ... }`): emitted from 7 sites in
   `shortcut/mod.rs` (`keyboard_implementation` incl. `reset_bindings`, `debug_mode`,
   `start_hidden`, `autostart_enabled`, `update_checks_enabled`, `show_whats_new_on_update`,
   `whats_new_last_seen_version`). **No frontend listener exists for this event** (grep of
   `src/` finds none) — it is currently dead air. Backend-side binding resets or the
   HandyKeys->Tauri fallback are not reflected in an open settings UI until something else
   triggers a refetch.

### Model store (`src/stores/modelStore.ts`)

Owns the model catalog and download lifecycle (other subsystem), but is part of settings
plumbing for `selected_model`:

- `selectModel` -> `set_active_model` -> `commands/models.rs (switch_active_model)`: claims a
  loading guard, persists `selected_model` + `onboarding_completed = true` *before* loading so UI
  event handlers see the right value, loads the model (skipped when
  `model_unload_timeout == immediately`, emitting `model-state-changed` /
  `selection_changed` instead), and **reverts the persisted selection on load failure**.
- `deleteModel` of the active model unloads it and blanks `selected_model`.
- Listens to: `model-download-progress`, `model-download-complete`, `model-download-failed`
  (toast), `model-verification-started/completed`, `model-extraction-started/completed/failed`,
  `model-download-cancelled`, `model-deleted`, `model-state-changed`, `models-updated`.

## Frontend store details

### `src/stores/settingsStore.ts`
- zustand + `subscribeWithSelector`. State: `settings`, `defaultSettings`, `isLoading`,
  `isUpdating: Record<string, boolean>` (keys are setting names, `binding_<id>`,
  `post_process_<type>:<provider>`), `audioDevices`, `outputDevices`, `customSounds`,
  `postProcessModelOptions: Record<providerId, string[]>` (session-only cache, never persisted).
- `settingUpdaters` — the single dispatch table from settings key -> command. Keys deliberately
  absent: `bindings` (use `updateBinding`), `selected_model` (use `modelStore.selectModel`).
  Keys absent with **no path at all** through `updateSetting`: `settings_schema_version`,
  `onboarding_completed`, `model_unload_timeout` (component calls
  `commands.setModelUnloadTimeout` directly — `src/components/settings/ModelUnloadTimeout.tsx`),
  `keyboard_implementation` (component calls `commands.changeKeyboardImplementationSetting`
  directly), `custom_filler_words` (no command exists), and the post-process collections
  (handled by dedicated store actions `setPostProcessProvider`, `updatePostProcessBaseUrl`,
  `updatePostProcessApiKey`, `updatePostProcessModel`, plus prompt commands called from
  components).
- `updatePostProcessBaseUrl` is sequenced: persist base URL -> reset stored model to `""` ->
  clear cached model options -> single `refreshSettings`.
- Device-name impedance: UI uses `"Default"`, updater sends `"default"`, Rust persists `None`.

### `src/hooks/useSettings.ts`
Pure convenience wrapper; calls `store.initialize()` on first mount while `isLoading` and
re-exports state and actions (also exposes `audioFeedbackEnabled`).

## Platform-specific behavior (macOS especially)

- Default shortcuts: macOS `option+space` / `option+shift+space`; Windows/Linux `ctrl+space` /
  `ctrl+shift+space` (`settings.rs (get_default_settings)`).
- `keyboard_implementation` defaults to `handy_keys` everywhere except Linux (`tauri`).
- `paste_method` defaults to `ctrl_v` on macOS/Windows, `direct` on Linux; `typing_tool` is
  Linux-only (macOS returns `["auto"]` from `get_available_typing_tools`).
- `overlay_style` defaults to `live` except Linux (`none`).
- Apple Intelligence provider is only added on macOS aarch64, and availability is deliberately
  NOT checked at startup (macOS 26.x beta SIGABRT when touching `SystemLanguageModel.default`
  early); it is checked on use via `check_apple_intelligence_available`.
- macOS permission choreography: `App.tsx` checks accessibility + microphone permission
  (`tauri-plugin-macos-permissions-api`) before deciding onboarding; `initialize_enigo` fails
  until accessibility is granted; audio device enumeration is deferred until after onboarding so
  the mic permission dialog doesn't fire on first paint.
- `app_language` default comes from the OS locale.
- Store file on macOS: `~/Library/Application Support/com.pais.handy/settings_store.json`
  (identifier `com.pais.handy`); portable mode is effectively a Windows feature but the code is
  platform-neutral.

## Fragile points & failure modes

1. **Corrupt settings file silently resets everything.** `settings.rs (get_settings /
   load_or_create_app_settings)`: any deserialization error -> overwrite the store with defaults.
   Only `load_or_create_app_settings` logs a `warn!` (`settings.rs:916`); `get_settings`'
   parse-failure branch (`settings.rs:951-955`) discards the error and resets with no logging at
   all. User loses shortcuts, API keys, prompts, model selection with zero UI feedback.
   One bad enum value written by a future version downgrade is enough to trigger this.
2. **Whole-struct read-modify-write with no lock.** Every setter does
   `get_settings -> mutate -> write_settings`. Two concurrent commands (or a command racing the
   transcription flow's own writes, e.g. `switch_active_model`) can lose one of the updates —
   last writer wins on the entire blob.
3. **Store panics.** `.expect("Failed to initialize store")` in all three settings.rs accessors;
   `serde_json::to_value(...).unwrap()` on write; `settings.rs (get_stored_binding)` does
   `.unwrap()` on the binding id (panics if a caller asks for an id missing from the store).
4. **`settings-changed` has no listener.** Backend-side mutations it announces (binding resets on
   keyboard-impl switch, HandyKeys->Tauri fallback persisted at startup, debug/autostart toggles
   from non-UI paths) never update an open UI; the UI only resyncs on `model-state-changed` or a
   window reload.
5. **Silent-write failures.** tauri-plugin-store's debounced auto-save discards errors
   (`let _ = self.trigger_auto_save()`); a full disk or permission error means settings appear
   saved but are not, with no log and no UI feedback.
6. **Silent coercion of invalid values.** String-typed setters (`sound_theme`, `paste_method`,
   `overlay_position`, `overlay_style`, `typing_tool`, `auto_submit_key`,
   `clipboard_handling`, `keyboard_implementation`) map unknown strings to a default with only a
   `warn!`; the command still returns `Ok`, so the frontend's optimistic value can differ from
   what was persisted.
7. **Optimistic UI with console-only errors.** `settingsStore.updateSetting` rolls back state but
   reports failures only to the console — the user sees a toggle snap back with no explanation.
   (Contrast: `modelStore` shows toasts for download failures; `updateBinding` rethrows so
   shortcut inputs can display errors.)
8. **Three-way wiring for every new setting.** Adding a key requires: Rust field + serde default,
   a `change_*` command registered in `lib.rs (collect_commands!)`, and a `settingUpdaters`
   entry. Missing the last one yields only `console.warn("No handler for setting")` and an
   in-memory-only change that silently reverts on next refetch.
9. **bindings.ts only regenerates in debug builds.** Adding/renaming a command without running a
   dev build leaves the committed `src/bindings.ts` stale; release builds will happily ship the
   mismatch.
10. **Secrets in plaintext.** `post_process_api_keys` (`SecretMap`) is redacted only in `Debug`
    formatting; the store JSON on disk contains raw keys.
11. **Read-triggered writes.** `get_settings` may write (migrations + provider back-fill) on any
    command; benign because idempotent, but surprising — e.g. merely opening the settings UI can
    rewrite the file, and a half-broken store gets "repaired" (or reset, see #1) on read.
12. **FE/disk representation drift.** The store normalizes device fields `null -> "Default"` in
    memory; comparisons against `defaultSettings` (which keeps `null`) or naive deep-equality
    checks will mismatch.
13. **Autostart and shortcut side effects ignore errors.** `change_autostart_setting` ignores
    `enable()/disable()` results; `change_post_process_enabled_setting` ignores shortcut
    (un)registration failures — the persisted flag can disagree with actual system state.
