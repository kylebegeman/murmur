# Recording Overlay Subsystem

Status: scratchpad architecture map, produced by reading source on 2026-07-04. No code was modified.

## Purpose

The recording overlay is a small, always-on-top, transparent, non-focusable window that gives the user visual feedback during dictation: a pulsing dot + live waveform while recording, a spinner + label while transcribing/post-processing, and (in "Live" style with a streaming-capable model) a panel that shows the live transcript as it is decoded, split into a stable committed prefix and a volatile tentative suffix. At the end of every non-cancelled take it shows a result row reporting what happened to the words (inserted / copied / failed / â€¦) with a Copy recovery action on failures. It also hosts the only always-visible cancel affordance (the X button).

One native window (label `recording_overlay`) is created hidden at app boot and reused for every state; Rust resizes/repositions/shows/hides it and drives the React UI inside it entirely via events.

## Key functions / structs / components

### Rust â€” `src-tauri/src/overlay.rs`

- `RecordingOverlayPanel` (macOS only) â€” generated by `tauri_nspanel::tauri_panel!` macro with `can_become_key_window: false`, `is_floating_panel: true`.
- Constants:
  - `OVERLAY_WIDTH = 256.0`, `OVERLAY_HEIGHT = 46.0` (compact window, logical points)
  - `OVERLAY_STREAM_WIDTH = 400.0`, `OVERLAY_STREAM_HEIGHT = 120.0` (Live/streaming window; actual card is 394x118 per the comment at `overlay.rs:48` â€” the CSS content width `--ov-open-w` is 392px plus a 1px border each side, with no `box-sizing: border-box` reset in the overlay CSS chain)
  - `OVERLAY_TOP_OFFSET`: 46.0 on macOS (clears menu bar), 4.0 on Windows/Linux
  - `OVERLAY_BOTTOM_OFFSET`: 15.0 on macOS, 40.0 on Windows/Linux (clears taskbar)
- `overlay_dimensions(state)` â€” `"streaming"` gets the big window; every other state gets the compact one.
- `create_recording_overlay(app_handle)` â€” two implementations:
  - non-macOS: `WebviewWindowBuilder` with `always_on_top(true)`, `skip_taskbar(true)`, `transparent(true)`, `focusable(false)`, `visible(false)`, `accept_first_mouse(true)` (so the cancel X is clickable without focusing), `closable(false)`, `decorations(false)`, `shadow(false)`. Optional portable-mode `data_directory` (`crate::portable::data_dir()`). On Linux, then tries `init_gtk_layer_shell`.
  - macOS: `PanelBuilder::<_, RecordingOverlayPanel>` (tauri_nspanel) â€” `PanelLevel::Status`, `StyleMask::empty().borderless().nonactivating_panel()`, `no_activate(true)`, `CollectionBehavior::new().can_join_all_spaces().full_screen_auxiliary()` (visible on all Spaces and over full-screen apps), transparent, no shadow. Window stays registered so `get_webview_window("recording_overlay")` keeps working. `panel.hide()` immediately after build.
- `show_overlay_state(app_handle, state)` â€” the single show path: bumps `OVERLAY_SESSION` (so any pending delayed hide from an earlier take no-ops), then no-op when `overlay_style == None`; else `set_size` â†’ `calculate_overlay_position` â†’ `set_position` â†’ `show()` â†’ (Windows) `force_overlay_topmost` â†’ `emit("show-overlay", state)` on the overlay window. Logs per-step timings at debug.
- Public wrappers: `show_recording_overlay` ("recording"), `show_streaming_overlay` ("streaming"), `show_transcribing_overlay` ("transcribing"), `show_processing_overlay` ("processing").
- `hide_recording_overlay(app_handle)` â€” the immediate-teardown hide (cancellation, recording errors): delegates to `hide_overlay_for_session` with the current session. `hide_overlay_for_session(app_handle, session)` gates **both** the `hide-overlay` emit and the delayed native `window.hide()` (300 ms later) on `OVERLAY_SESSION` still equalling `session`, so a hide from take N can never fade take N+1 (closes the emit-before-guard TOCTOU the earlier version had).
- `show_result_overlay(app_handle, event, hold_ms, expected_session)` â€” shows a dictation result (inserted / copied / failed / â€¦). `expected_session` is the `OVERLAY_SESSION` captured by the take when it last touched the overlay (for the paste path, captured in `actions.rs` _before_ the main-thread paste is scheduled â€” the paste runs after the coordinator returns to Idle, so a newer take can take over the overlay while it runs). If the session has advanced, the result is stale UI and is dropped entirely (no emit, no hide). Otherwise it emits the typed `DictationResultEvent` (tauri_specta broadcast, wire name `dictation-result-event`; defined in `actions.rs`) **regardless of `overlay_style`** â€” any window may listen and the "never silent after a recording" contract must hold even on Linux where the overlay defaults to `None`. Only the visible overlay's timed hide (via `hide_overlay_for_session` after `hold_ms`) is gated on the overlay being enabled. Hold constants live in `actions.rs`: success 1400 ms, info 1800 ms, failure 8000 ms.
- `OVERLAY_SESSION: AtomicU64` â€” monotonic id of the current overlay "visual session", bumped on every `show_overlay_state`. Delayed hides (both the 300 ms fade hide and the result hold) capture the id when scheduled and no-op if a newer show happened in the meantime.
- `update_overlay_position(app_handle)` â€” repositions without showing; uses `current_overlay_logical_size` (falls back to compact constants) so centering is right for both layouts. Called when position/style settings change.
- Positioning helpers:
  - `get_monitor_with_cursor` â€” `input::get_cursor_position` (enigo; other subsystem) then matches against `available_monitors()`, dividing monitor physical position/size by `scale_factor()` to normalize to logical coordinates (enigo may return logical on macOS). Falls back to `primary_monitor()`.
  - `calculate_overlay_position(app, w, h)` â€” horizontal center of the cursor's monitor; y from `settings.overlay_position` (Top/Bottom) plus the per-platform offset. Deliberately uses monitor bounds (not `work_area()`, wrong on macOS with negative monitor positions) and `LogicalPosition` (`PhysicalPosition` converts with the wrong scale factor cross-monitor).
- Level emission:
  - `OVERLAY_ENABLED: AtomicBool` (defaults false) + `update_overlay_enabled_cache(enabled)` â€” caches "overlay_style != None" so the ~24â€“30 Hz audio callback does one relaxed atomic load instead of a Tauri store read.
  - `emit_levels(app_handle, levels)` â€” skips when disabled (WebKit memory-growth workaround, issue #1279: even a hidden overlay's WebKit subprocess accumulates allocations per event); otherwise `app_handle.emit_to("recording_overlay", "mic-level", levels)` â€” `emit_to` (not broadcast) halves per-callback `eval_script` work.
- Platform helpers: `update_gtk_layer_shell_anchors`, `env_flag_enabled` ("HANDY_NO_GTK_LAYER_SHELL"), `init_gtk_layer_shell` (Layer::Overlay, KeyboardMode::None, exclusive_zone 0), `force_overlay_topmost` (Win32 `SetWindowPos(HWND_TOPMOST, SWP_NOMOVE|SWP_NOSIZE|SWP_NOACTIVATE|SWP_SHOWWINDOW)` on the UI thread).

### Rust â€” event/data types (`src-tauri/src/managers/transcription.rs`)

- `StreamTextEvent { committed: String, tentative: String }` â€” `tauri_specta::Event`; wire name `stream-text-event`. `committed` is the append-only, flicker-free prefix; `tentative` is the suffix the model may still rewrite.
- `StreamPhase` â€” `listening` | `working` (serde lowercase). Rust never emits `listening`; the frontend starts there and Rust only emits the transition to `working`.
- `StreamWorkKind` â€” `transcribing` | `polishing` (lowercase).
- `StreamPhaseEvent { phase, kind: Option<StreamWorkKind> }` â€” wire name `stream-phase-event`; `kind` omitted from JSON when `None`, present only when `phase == working`.
- `TranscriptionManager::emit_stream_working(kind)` / `emit_stream_text(committed, tentative)` â€” broadcast via `.emit(&self.app_handle)` (all windows; only the overlay listens).
- `run_stream_worker` â€” on each fed audio frame, if `update.committed_changed || update.tentative_changed`, reads `stream.text()` and calls `emit_stream_text` (transcription.rs ~line 945).

### Frontend â€” `src/overlay/RecordingOverlay.tsx`

- `RecordingOverlay` (default export) â€” the entire UI. Internal state:
  - `isVisible: boolean` â€” drives fade in/out.
  - `state: OverlayState = "recording" | "streaming" | "transcribing" | "processing"` â€” payload of `show-overlay`.
  - `levels: number[]` â€” first `WAVE_BARS = 9` of 16 exponentially smoothed FFT buckets (`prev*0.7 + target*0.3`, kept in `smoothedLevelsRef` across renders and sessions).
  - `streamText: StreamTextEvent` â€” `{ committed, tentative }`.
  - `phase: StreamPhase` (starts `"listening"`), `workKind: StreamWorkKind` (starts `"transcribing"`).
  - `elapsed: number` â€” seconds; ticks only while `state === "streaming" && isVisible` and no result is showing (the take is over once a result arrives; compact recording pill has no timer).
  - `session: number` â€” bumped on every `show-overlay: "streaming"`; used as React `key` on the card so each Live session remounts fresh (replays pop-in, never animates from the previous panel's size).
  - `position: "top" | "bottom"` â€” fetched from `commands.getAppSettings()` on every `show-overlay` (fetch failure silently keeps previous value).
  - `overflowing: boolean` â€” true once live text overflows the 64px cap; gates the top-edge fade for top placement.
  - `capRef` / `pinnedRef` â€” scroll-back: auto-scroll sticks to the newest line while the user is within ~16 px of the bottom (`handleStreamScroll`); scrolling up pauses auto-follow; each new session re-pins.
  - `result: DictationResultEvent | null` â€” terminal outcome of the take, set by `dictation-result-event` and cleared on every `show-overlay` (a new take always clears the previous outcome). While set, the row area shows the result row instead of the waveform/spinner â€” in **both** the Live panel and the compact pill.
  - `copiedTranscript: boolean` â€” whether the result row's Copy recovery has been used for the current result; reset whenever `result` changes or a new show arrives.
- Building blocks: `waveform` (9 `<i>` bars, height `max(3, min(18, 3 + v^0.7 * 15))` px), `cancelBtn` (`.sx`, calls `commands.cancelOperation()`), `listeningRow(showTimer, showCancel)` (dot | waveform | timer+cancel 3-zone grid), `workingRow(label, showCancel)` (spinner | label | cancel), `resultRow(r)` (status icon ok/warn/err | outcome label | Copy recovery + dismiss). Result labels come from `RESULT_LABEL_KEYS` (`overlay.result.*` i18n keys, en only â€” other locales fall back) and tint from `RESULT_KIND` (ok = delivered somewhere, warn = nothing to deliver, err = actual failure). The Copy button renders when a transcript exists and the outcome isn't `inserted`/`copied`; it calls `commands.copyTranscript(transcript)` and flips to "Copied" (`copiedTranscript`) on success. The dismiss X (the shared `cancelBtn`) appears only for `err` outcomes.
- Rendering:
  - `state === "streaming"` (Live): `open = hasText` (committed or tentative non-empty) â†’ `.scard.open` (392 px panel). The panel stays open while finalizing so the transcript doesn't collapse; `collapsed = working && !hasText` â†’ small working pill (`.scard.working`). Committed renders in `.committed` (~94% text tone, plus trailing space), tentative in `.tentative` (~58% tone, so the still-forming tail reads as provisional), blinking `.scaret` shown only while not working. Working label: `overlay.processing` when `workKind === "polishing"`, else `overlay.transcribing` (i18n keys).
  - Other states (Minimal pill): one row at a time â€” `listeningRow(false, true)` when recording, `workingRow` when `state` is `transcribing`/`processing`; the pill width-animates between `--ov-rest-w` (172) and `--ov-work-w` (216). Fade via `.ov-fade.show`.
  - In both layouts, a non-null `result` takes precedence over the working/listening row (`resultRow` replaces it); the compact pill also keeps its wide `cworking` state while a result shows. Rust hides the window after the outcome's hold, so the row's lifetime is Rust-driven.
  - `dir` attribute from `getLanguageDirection(i18n.language)` (`src/lib/utils/rtl.ts`) for RTL locales.

### CSS â€” `src/overlay/RecordingOverlay.css`

- Geometry single source of truth (`:root`): `--ov-rest-w: 172px`, `--ov-pill-w: 184px`, `--ov-work-w: 216px`, `--ov-open-w: 392px`, `--ov-base-h: 40px`, `--ov-cap-max-h: 64px`. Comment contract: keep `overlay.rs` window constants â‰Ą these.
- `.ov-stage` â€” fixed, inset 0, transparent; `align-items: flex-end` (bottom) / `.top â†’ flex-start`. The card is anchored flush to the screen-edge side and centered horizontally, so window height doesn't affect where the card sits â€” only the Rust offsets do. Pop animations scale from that edge (matters for the 4 px top offset on Windows/Linux).
- `.scard` â€” shared card; `scard-pop` mount animation (460 ms), width/border-radius transitions for the pillâ†’panel morph; `.leaving` fades out (240â€“300 ms, matching Rust's 300 ms hide delay); `.open`, `.working`, `.compact`, `.compact.cworking` width states.
- `.stext` â€” grid-rows `0fr â†’ 1fr` trick for auto-height open; `.stext-cap` is the max-64px scrolling live-text region with a top fade mask, hidden scrollbars, bottom-anchored `<p>` (`margin: auto 0 0`). `.ov-stage.top` flips the card to `column-reverse` (pill above text), moves padding, and only applies the top fade when `.overflowing`.
- Theming pulls from `src/styles/theme.css` tokens (`--color-background`, `--color-logo-primary`, etc.); auto dark mode via `prefers-color-scheme`.

## Runtime flow

1. **Boot** â€” `lib.rs (setup)` â†’ `initialize_core_logic` â†’ `overlay.rs (create_recording_overlay)`: window/panel created hidden. `lib.rs` seeds `update_overlay_enabled_cache(overlay_style != None)`. The overlay webview loads `src/overlay/index.html`, mounts `RecordingOverlay`, registers listeners for `show-overlay`, `hide-overlay`, `mic-level`, `stream-text-event`, `stream-phase-event`, `dictation-result-event`.
2. **Record start** â€” `actions.rs (TranscribeAction::start)`: based on `overlay_style` and whether the model advertises streaming: `Live` + streaming model â†’ `show_streaming_overlay` ("streaming", window resized 400x120); `Live` (non-streaming model) or `Minimal` â†’ `show_recording_overlay` ("recording", 256x46); `None` â†’ nothing. `show_overlay_state` sizes, positions (cursor's monitor, top/bottom setting), shows, emits `show-overlay`.
3. **Frontend on `show-overlay`** â€” syncs i18n language, refetches `overlay_position` via `getAppSettings`, sets `state`; clears any previous `result` (and `copiedTranscript`); for "recording"/"streaming" clears `streamText`; for "streaming" also resets phase to `listening`, `workKind` to `transcribing`, `elapsed` to 0 and bumps `session` (card remount â†’ pop-in). Sets `isVisible = true`.
4. **While recording** â€” audio recorder's level callback (`managers/audio.rs (create_audio_recorder)` â†’ `audio_toolkit/audio/recorder.rs`, 16-bucket FFT over a ~33 ms window) â†’ `utils::emit_levels` â†’ `mic-level` to the overlay window only; frontend smooths and renders 9 bars. In streaming mode, the transcription worker emits `stream-text-event` on every committed/tentative change; the Live pill sculpts open into the panel on first text.
5. **Record stop** â€” `actions.rs (TranscribeAction::stop)`: if `(OverlayStyle::Live, is_streaming)` â†’ `tm.emit_stream_working(Transcribing)` (`stream-phase-event {phase:"working", kind:"transcribing"}`) â€” panel stays up, spinner replaces waveform; otherwise â†’ `show_transcribing_overlay` (re-emits `show-overlay: "transcribing"`, resizes to compact). If post-processing is enabled: Live style â†’ `emit_stream_working(Polishing)`, else â†’ `show_processing_overlay`.
6. **Finish/cancel/error** â€” non-cancelled terminal paths now call `utils::show_result_overlay` with a `DictationResultEvent` instead of hiding immediately: paste success (`inserted` / `copied` / `not_delivered`), no speech (`no_speech`), empty text (`empty_transcript`), transcription error (`transcription_failed`, also emits the `transcription-error` toast to the main window), paste error (`paste_failed`, also `paste-error` with a string detail). The overlay renders the result row; Rust hides after an outcome-dependent hold (success 1.4 s / info 1.8 s / failure 8 s) unless a new take started (session guard). The remaining paths still call `utils::hide_recording_overlay` directly and hide immediately: cancellation checks, recording-start failure (`recording-error`), and `utils.rs (cancel_current_operation)` (triggered by the overlay's own X button via `cancel_operation` command, or the Esc cancel shortcut). Frontend gets `hide-overlay` â†’ `isVisible = false` â†’ fade-out; Rust hides the native window 300 ms later (session-guarded).
7. **Settings changes** â€” `shortcut/mod.rs (change_overlay_position_setting)` writes the setting and calls `update_overlay_position`; `(change_overlay_style_setting)` writes, refreshes the `OVERLAY_ENABLED` cache, and repositions. The window is never recreated.

## Settings keys

- `overlay_style` â€” `"none" | "minimal" | "live"` (`settings.rs (OverlayStyle)`). Default: `live` everywhere except Linux, where it is `none` (`default_overlay_style`). Owns overlay visibility entirely; gates both `show_overlay_state` and `emit_levels`.
- `overlay_position` â€” `"top" | "bottom"` (`settings.rs (OverlayPosition)`); serde alias maps a legacy stored `"none"` onto `Bottom`. Default `bottom`. Read on every show (Rust y-placement) and on every `show-overlay` (frontend layout flip).
- Migration (`settings.rs` ~line 1012): when a stored settings blob has no `overlay_style`, legacy `overlay_position: "none"` becomes `overlay_style: "none"`, anything else becomes `"live"`.
- `app_language` â€” read (not written) by the overlay's i18n sync for the working labels and RTL direction.

## Platform-specific behavior

### macOS

- The overlay is an **NSPanel**, not a regular window: `tauri_nspanel` `PanelBuilder` with `PanelLevel::Status`, non-activating borderless style mask, `can_become_key_window: false`, `is_floating_panel: true`. Collection behavior `can_join_all_spaces + full_screen_auxiliary` keeps it visible across Spaces and over full-screen apps without switching Spaces.
- Offsets: top 46 pt (below menu bar/notch area), bottom 15 pt.
- Positioning notes baked into `calculate_overlay_position`: avoid `work_area()` (wrong for negative monitor origins on macOS) and use `LogicalPosition` (physical coords are converted with the current monitor's scale factor, breaking cross-monitor moves). `enigo`'s cursor position is logical on macOS (`NSEvent::mouseLocation`), hence the divide-by-scale normalization in `get_monitor_with_cursor`.
- Creation failure mode differs: if `calculate_overlay_position` returns `None` at boot (no cursor monitor and no primary monitor), the panel is **never created and nothing is logged** on macOS; build errors are logged at `error!`.

### Windows

- Regular Tauri window plus `force_overlay_topmost`: raw Win32 `SetWindowPos(HWND_TOPMOST, â€¦, SWP_NOACTIVATE)` on the UI thread after every show, because Tauri's `always_on_top` can be overridden by other topmost windows.
- Offsets: top 4, bottom 40 (taskbar). `GetCursorPos` logical-vs-physical depends on the process DPI-awareness context â€” the divide-by-scale normalization is described as safe but is heuristic.

### Linux

- `overlay_style` defaults to `none` (overlay hidden, `mic-level` suppressed) â€” chosen because of the WebKitGTK memory growth issue (#1279).
- When shown: GTK Layer Shell (`Layer::Overlay`, `KeyboardMode::None`, exclusive zone 0) with Top/Bottom edge anchors updated from settings on each show (`update_gtk_layer_shell_anchors`); escape hatch env var `HANDY_NO_GTK_LAYER_SHELL`. Falls back to a regular window when layer shell is unsupported (e.g. GNOME Wayland), where `set_position` may not work. Monitor detection is allowed to fail on creation (anchors don't need coordinates).

## Fragile points and failure modes

1. **FIXED â€” 300 ms delayed-hide race** â€” historically `overlay.rs (hide_recording_overlay)` hid the window 300 ms later unconditionally, so a new recording started within that window got its freshly shown overlay blanked while the frontend believed it was visible. Now every show bumps `OVERLAY_SESSION`, and both delayed hides (the 300 ms fade hide and the result-hold hide in `show_result_overlay`) capture the session id when scheduled and no-op if a newer show happened.
2. **Silent no-op if the window was never created** â€” every show/hide/update goes through `get_webview_window("recording_overlay")` and silently does nothing on `None`. On macOS a boot-time positioning failure means the panel is never created (see above) and the user simply never sees an overlay, with zero feedback and (on macOS) zero logging for the position-miss case; on Windows/Linux creation failure is only a `debug!` line.
3. **All window operations ignore errors** â€” `set_size`, `set_position`, `show`, `hide`, and both `emit` calls are all `let _ =`. A failed show or a failed `show-overlay` emit leaves native window and React state out of sync (e.g. window visible but content at opacity 0, or vice versa).
4. **First-show vs. webview-load race** â€” `show-overlay` is emitted immediately after `window.show()`. Listeners are registered asynchronously in a `useEffect`; if the overlay webview is still loading (slow machine, dev server), the event is dropped: the window is visible but the stage stays at opacity 0 (invisible), and `mic-level`/stream events for that session are also missed. Normally masked because the window is created at boot, long before the first recording.
5. **Listener cleanup never runs** â€” `RecordingOverlay.tsx (setupEventListeners)` returns its unlisten closure from an async function inside `useEffect`, so the effect's cleanup is undefined and the unlisteners are never invoked. Harmless today (component never unmounts; the window lives for the app's lifetime) but under React 18 StrictMode dev double-invoke this registers every listener twice (double state sets, benign) â€” a trap for future refactors.
6. **CSS/Rust geometry coupling by comment only** â€” window sizes in `overlay.rs` must be at least the CSS card footprints (`--ov-work-w`, `--ov-open-w`, heights). Nothing enforces it; growing a CSS width without touching `OVERLAY_*` constants clips the card. Murmur polish work will hit this immediately.
7. **`overlay_position` fetched per show over IPC** â€” a failed `getAppSettings` on show is swallowed (`catch {}`) and silently keeps the previous placement; there is also a frame-level chance of showing with a stale top/bottom layout while the async fetch resolves after `setIsVisible(true)`.
8. **Streaming panel never resizes on phase change** â€” the stop path in Live mode keeps the 400x120 window and only flips phase to `working`. Fine by design, but if `overlay_style` changes mid-recording, mixed states are possible (hide always works regardless of style, by design; show respects the new style).
9. **`OVERLAY_ENABLED` starts `false`** â€” `mic-level` is suppressed until `lib.rs (setup)` seeds the cache; a recording started impossibly early would show a dead waveform. Also, style set to `none` mid-recording freezes the bars at their last values (levels stop arriving; `smoothedLevelsRef` holds state and is never cleared on hide or new session â€” stale bar heights can flash at the start of the next session until the first `mic-level` arrives).
10. **Cursor-position heuristics** â€” `get_monitor_with_cursor` divides physical monitor bounds by scale to compare with enigo's cursor coordinates; the code itself documents that Windows behavior depends on DPI-awareness context. Wrong matches put the overlay on the wrong monitor; a total miss falls back to the primary monitor.
11. **Rust never emits `phase: "listening"`** â€” the frontend resets the phase locally on each `show-overlay: "streaming"`. Any future Rust path that shows the streaming overlay without a fresh `show-overlay` (or emits `working` before the show event is processed) will desynchronize the phase.
12. **RESOLVED â€” `tentative` now reads as provisional** â€” `.committed` renders at ~94% text tone and `.stext-cap .tentative` at ~58%, so the still-forming tail is visibly lighter than the locked-in prefix (the caret still marks liveness). Purely a CSS change; the committed/tentative payload split is untouched.
13. **Elapsed timer is UI-local** â€” it counts seconds from the `show-overlay` event, not from actual capture start, and only exists in the Live panel; the compact pill renders no timer even though `listeningRow` supports one (`showTimer=false`).
